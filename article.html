Every year, a few weeks before Roland Garros, tennis fans argue about who is going to win. This project turns those intuitions into numbers — a full probability matrix for 128 players across all 7 rounds of the tournament.

The goal is to answer a deceptively simple question: given everything we know about each player heading into Roland Garros 2026, what is the probability that each one wins the title? And not just the title — what are the odds each player survives the first round, reaches the quarter-finals, makes the final?

To do that, we build a four-stage pipeline running entirely from the command line, with a Streamlit dashboard as the final output.

Why This Approach?

Before diving in, it is worth explaining the philosophy — several alternative approaches exist, and the choices made here are deliberate.

We model individual matches, not aggregate outcomes directly

Some tennis prediction systems jump straight to tournament winner odds using betting markets or simple ranking heuristics. We instead estimate the probability of each individual match, then compound those across seven rounds. This matters because a player who wins 65% of their matches still only wins a 7-round bracket about 18–25% of the time. By modelling at the match level, we see exactly how uncertainty accumulates — and we can ask counterfactual questions like "what if Alcaraz draws Sinner in the quarter-finals?"

We use machine learning, not just Elo

The core predictor is XGBoost trained on 25 years of ATP matches. Pure Elo is elegant but ignores surface specialisation, recent form, head-to-head history, and age trajectories. XGBoost combines all of these into one calibrated probability estimate.

Design Choice We use a single model trained on all surfaces, not separate per-surface models. Surface specialisation is captured as a feature (surface_elo_diff) rather than a model split. This avoids training on too few clay-only samples to fit a reliable model.

We simulate the bracket, not just rank players

The natural temptation after computing clay ratings is to rank players and call the top one the favourite. But tournaments are not played on paper. Draw luck matters enormously. Monte Carlo simulation makes draw luck explicit: we run 20,000 independent bracket realisations and count how often each player wins. This approach is directly analogous to probabilistic Premier League season simulators — except instead of league positions, we track tournament rounds.

Step 1

1_fetch_data.py — 25 Years of ATP History

All match data comes from Jeff Sackmann's open tennis_atp repository on GitHub, which contains ATP match records dating back to 1968. We use 2000–2024, giving approximately 74,000 matches. Each year is a separate CSV file downloaded once and cached locally. 

python 1_fetch_data.py

# Downloading matches: 100%|██████| 25/25
# Loaded 74,906 matches (2000–2024)
# Loaded 55,193 players

Each record contains: winner, loser, surface (Hard / Clay / Grass), tournament level (Grand Slam, Masters, ATP 250…), the score, and both players' ATP rankings at match time.

Design Choice We start at year 2000, not earlier. Pre-2000 rankings are sparser and the player population differs significantly from the modern era. Twenty-five years gives ~75,000 matches — more than enough to train a robust model while keeping the data relevant.
Step 1b

fetch_recent.py — Supplementing with 2026 Clay Data

JeffSackmann's repository is updated manually and typically lags the current date by several weeks. For Roland Garros 2026, that means the clay build-up season — Monte-Carlo, Barcelona, Madrid, Rome — may only be partially available. We supplement it with tennis-data.co.uk, which publishes results within days of each match.

The critical difference is the player identifier. JeffSackmann uses stable numeric IDs like 207989 (Jannik Sinner). tennis-data uses abbreviated name strings: "Sinner J.". Before we can merge the two datasets, we must resolve every tennis-data name into the corresponding JeffSackmann ID.

The Name Resolution Problem

tennis-data names follow the pattern "Lastname F." — and sometimes "Lastname F.M." for double initials, or "Two Part Last C." for hyphenated surnames. The resolver handles all cases in three fallback layers: 

"Sinner J."        → parse → (last="sinner",  initial="j")  → exact match → 207989
"Cerundolo J.M."   → parse → (last="cerundolo", initial="j")  → exact match → 126610
"Ugo Carabelli C." → parse → (last="ugo carabelli", initial="c") → exact match
"Unknwon P."       → parse → fuzzy match on last name → best candidate or -1
Design Choice — Three-Layer Resolution 1. (last_name, first_initial) exact match — works for 95%+ of players.
2. Last-name-only — for surnames unique in the ATP, such as "Djokovic".
3. Fuzzy match (cutoff 0.75) — catches accents and slight spelling differences, e.g. "Müller" vs "Muller". Any match that fails all three layers is dropped with a warning rather than guessing — we never inject a wrong player ID into the training data.

Deduplication

Some matches appear in both sources — JeffSackmann may have the early rounds of Madrid while tennis-data has the full draw. After resolving IDs, we build a set of (winner_id, loser_id) pairs already present in the JeffSackmann data and exclude any tennis-data record that matches. This ensures each match is counted exactly once.

Step 2

2_build_features.py — Engineering Player Ratings

Raw match results are not useful to a model. We need features that capture each player's current strength just before each match is played. Four modules compute these features under a strict no look-ahead constraint: every value is computed using only matches that occurred before the one being labelled.

Elo Ratings

The classic chess rating system applied to tennis. Every player starts at 1500. After each match, the winner gains points proportional to the upset magnitude and the loser loses the same amount. We maintain two parallel Elo ratings per player: a global one and a surface-specific one.

P(A beats B) = 1 / ( 1 + 10 ( Elo_B − Elo_A ) / 400 )

The clay Elo is particularly important for Roland Garros — it accumulates evidence exclusively from clay-court matches and is the most stable long-run signal of clay court ability.

Head-to-Head Record

For each match we record how many times the winner has beaten the loser in all prior meetings. When two players have never met, this defaults to 0.5 — no information either way.

Rolling Surface Win Rate

Using a 365-day rolling window, we compute each player's win rate specifically on the current surface. A player on a 12-month hot streak on clay scores higher here than their long-run Elo suggests.

Recent Form

Win rate across the last 20 matches on any surface. This captures whether a player is currently healthy and confident — or going through a rough patch.

Step 3

3_train_model.py — Learning to Predict Matches

Dataset Symmetrisation

Each match record (winner, loser) is duplicated as (loser, winner) with the label flipped from 1 to 0. This doubles the training set from ~74,000 to ~148,000 examples and forces the model to learn symmetrically — it cannot exploit the fact that the "winner" column always won.

Train / Validation Split

We train on 2000–2021 and validate on 2022–2024. This is a temporal split, not a random split. Random splits would leak future information: a match from 2023 in the training set would have Elo values that already absorbed 2023 results.

XGBoost Model

500 gradient-boosted trees, depth 4, learning rate 0.03. The logistic objective means probabilities are meaningful — 0.65 means the model genuinely believes there is a 65% chance player 1 wins, not just a ranked score.

# Val   log-loss : 0.608
# Val   Brier    : 0.211   ← well-calibrated probabilities

Live Player State

After training, we replay all 74,000 matches chronologically through a PlayerStateTracker. This object maintains, for each of the 2,640 ATP players ever recorded, their current state: Elo ratings, ATP rank, clay win rate, form, and head-to-head records against every other player. This state is what powers predictions for any future matchup.

Design Choice The player state tracker is saved as data/processed/player_states.pkl. This means 4_simulate_rg2026.py does not need to replay 74,000 matches every time — it loads instantly and goes straight to the simulation. Each file acts as a checkpoint in the pipeline.
Step 4

4_simulate_rg2026.py — The Monte Carlo Tournament

Selecting the Field

We take the top 128 active ATP players by current ranking. These are the tournament entrants.

Clay Strength Scoring

Before building the draw, we rank all 128 players by a composite clay strength score. This is analogous to attack and defense parameters in Poisson football models — a single number that summarises how dangerous a player is on clay heading into the tournament.

Clay Strength  =  0.45 × Clay Elo (z-score)  +  0.30 × Clay Win Rate — 12 months (z-score)  +  0.25 × Overall Form — last 20 matches (z-score)
Design Choice Clay Elo gets the highest weight (45%) because it is the most stable signal over thousands of matches. Recent win rate (30%) captures current form on the surface. Overall form (25%) ensures a player on a hot streak is not penalised for playing on hard courts in the build-up. All three components are z-score normalised so they are on the same scale before combining.

Draw Construction

We replicate the official Grand Slam seeding rules:

  • Seed 1 → fixed position 1 (top of top half)
  • Seed 2 → fixed position 128 (bottom of bottom half)
  • Seeds 3–4 → drawn randomly to the two half-boundary slots
  • Seeds 5–8 → one drawn randomly per quarter
  • Seeds 9–16 and 17–32 → fill their respective sections
  • Remaining 96 unseeded players → placed randomly

Monte Carlo Simulation

We run 20,000 independent simulations of the full 128-player bracket. In each simulation:

  1. Start with the draw (bracket positions are the same in every simulation)
  2. For each first-round match, draw a random outcome from the match win probability
  3. The winner advances; repeat for Round 2 through the Final
  4. Record which player won the title, and which round each player reached

After 20,000 runs, each player's probability of reaching each round is the fraction of simulations where they reached it. With 20,000 runs, championship odds are stable to within roughly ±0.5 percentage points.

Design Choice — Match Probability Caching In a 128-player bracket, each unique pair of players is scored by the model at most once per simulation. But across 20,000 simulations the same pairings recur constantly. We cache all computed probabilities so each unique pair is evaluated only once. This reduces the per-simulation cost from 127 model evaluations to near zero after the first few hundred simulations, making 20,000 runs practical in minutes rather than hours.

What This Model Does Not Do

Being explicit about limitations matters as much as describing capabilities.

  • No injury or withdrawal modelling.
    If Sinner withdraws two days before the tournament, the model does not know. The draw would need to be rebuilt manually.
  • No match-level context.
    We do not model fatigue across consecutive matches, weather conditions, or scheduling gaps. Tennis is a physically brutal sport; back-to-back five-set matches are not captured.
  • No betting market calibration.
    Unlike Poisson football simulators, we do not blend our probabilities with bookmaker odds. The model is a pure statistical estimate from historical data — self-contained and fully interpretable, but not attempting to beat the market.
  • 2024 data cutoff.
    Training data ends at the 2024 season. To refresh the model, re-run 1_fetch_data.py once the 2025 CSV is available in the JeffSackmann repository, then re-run the full pipeline.

Each script is self-contained and runnable from the repo root with python <script>. The .pkl files act as checkpoints — you can re-run any single stage without re-running the whole pipeline.

Downloads one CSV per year from the JeffSackmann GitHub repository and concatenates them into a single DataFrame. Files already present in save_dir are read from disk rather than re-downloaded. Raises RuntimeError if no data could be fetched.

Downloads the full-year ATP Excel file from http://www.tennis-data.co.uk/{year}/{year}.xlsx and caches it locally. Returns the raw DataFrame with tennis-data column names (Winner, Loser, WRank, Surface, …).

In year=2026, save_dir="data/raw"
Out Raw DataFrame with tennis-data schema (~1,000 rows for a full season)
src/data/fetch_recent.py resolve_player_id(name, primary, secondary, all_last_names)

Maps a tennis-data abbreviated name string (e.g. "Sinner J.") to a JeffSackmann numeric player ID. Tries three strategies in order: (1) exact (last_name, first_initial) lookup, (2) unambiguous last-name-only lookup, (3) fuzzy last-name match with cutoff 0.75. Returns -1 if all strategies fail — the record is then dropped.

In name — raw string from tennis-data  |  primary, secondary — dicts from build_name_index()
Out int — JeffSackmann player_id, or -1

Converts a raw tennis-data DataFrame into JeffSackmann-compatible format. Filters by surface and completion status, resolves all player names to IDs, renames and casts every column to match the JeffSackmann schema. Drops rows where either player ID could not be resolved.

In raw tennis-data DataFrame  |  players_df from fetch_players()  |  surface_filter="Clay"
Out JeffSackmann-schema DataFrame — ready to pd.concat() with historical data
src/data/process.py clean_matches(df)

Standardises a raw match DataFrame: parses tourney_date to datetime, maps surface names to canonical labels (Hard / Clay / Grass / Carpet), fills missing ranks with 9999, flags retirements and walkovers as is_complete=False, sorts chronologically, and assigns a global match_idx. This is the single cleaning step that every downstream module depends on.

In Raw DataFrame from fetch_matches() or pd.concat([...])

Orchestrates the full feature-engineering pipeline: runs Elo, H2H, surface win rate, and form modules in order (each adding columns to the DataFrame) then caches the result as a pickle. On subsequent runs use_cache=True loads the pickle directly — saving 5+ minutes of computation.

src/features/builder.py symmetrize(df)

Doubles the training dataset by duplicating each match as two rows: one where player 1 is the winner (target=1) and one where the roles are swapped (target=0). Computes difference features (elo_diff, rank_diff, etc.) from the per-player columns. This is what the XGBoost model is trained on.

Fits an XGBoost binary classifier on the symmetrised DataFrame. Drops incomplete matches (retirements / walkovers) if complete_only=True, removes rows with NaN features, then trains and saves the model with joblib.dump(). Returns the fitted model object.

src/models/predict.py PlayerStateTracker.process_all(matches)

Replays all 76,000 matches in chronological order, maintaining a live state for each of the 2,640+ players: global Elo, surface Elo (one per surface), ATP rank, form deque (last 20 matches), rolling surface win rate, and head-to-head records against every opponent. After calling this, tracker.get_player_state(pid, surface) returns the player's state as of the last match in the dataset — i.e., right now.

src/models/predict.py MatchPredictor.predict(p1, p2, surface, h2h_winrate, h2h_total)

Takes two player state dicts (from tracker.get_player_state()), builds the 11-feature vector, and returns P(p1 beats p2) on the given surface. The complement 1 − p is P(p2 beats p1). Internally this is just one row passed through the XGBoost model's predict_proba().

In Two player state dicts  |  surface string  |  h2h win rate and total
Out float in [0, 1] — win probability for player 1
src/rg/clay_rating.py compute_clay_ratings(tracker, player_ids)

Computes a composite clay strength score for each player in the field. Pulls clay_elo, clay_winrate, and form from the tracker, z-score normalises each dimension across the 128-player field, then weights them 45 / 30 / 25. The output is sorted descending — index 0 is the top seed for draw purposes.

In Fitted PlayerStateTracker  |  list of 128 player IDs
Out DataFrame (128 rows) with columns: player_id, name, rank, clay_elo, clay_winrate, form, clay_strength
src/rg/draw.py create_rg_draw(player_ids, rng_seed)

Assigns 128 players to bracket positions following official Grand Slam seeding rules. Seeds 1 and 2 are fixed. Seeds 3–4 draw randomly to the half-boundary slots. Seeds 5–8 draw to quarter boundaries. Seeds 9–16 and 17–32 fill their sections. The remaining 96 unseeded players are shuffled into empty slots. The same rng_seed always produces the same bracket.

In 128 player IDs sorted by seeding (seed 1 first)  |  rng_seed=42
Out List of 128 player IDs in bracket order — match pairs are (draw[0],draw[1]), (draw[2],draw[3]), …
src/rg/simulator.py RGSimulator.simulate(draw, n_sim, seed, verbose)

Runs n_sim independent Monte Carlo simulations of the 128-player bracket. In each simulation, match outcomes are sampled from the XGBoost match probabilities. Round counts are accumulated per player; after all simulations the counts are divided by n_sim to give reach probabilities. Match probabilities are cached so each unique pair is evaluated only once across all 20,000 runs — making the simulation fast.

In draw — 128 player IDs in bracket order  |  n_sim=20_000
Out DataFrame (128 rows × 8 round columns: R128, R64, R32, R16, QF, SF, F, W) — saved to data/rg2026_results.pkl
