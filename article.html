Every year, a few weeks before Roland Garros, tennis fans argue about who is going to win. This project turns those intuitions into numbers — a full probability matrix for 128 players across all 7 rounds of the tournament.

The goal is to answer a deceptively simple question: given everything we know about each player heading into Roland Garros 2026, what is the probability that each one wins the title? And not just the title — what are the odds each player survives the first round, reaches the quarter-finals, makes the final?

To do that, we build a four-stage pipeline running entirely from the command line, with a Streamlit dashboard as the final output.

Why This Approach?

Before diving in, it is worth explaining the philosophy — several alternative approaches exist, and the choices made here are deliberate.

We model individual matches, not aggregate outcomes directly

Some tennis prediction systems jump straight to tournament winner odds using betting markets or simple ranking heuristics. We instead estimate the probability of each individual match, then compound those across seven rounds. This matters because uncertainty multiplies across rounds: even a dominant player who wins 80% of individual matches needs to win seven in a row, and 0.80 × 0.80 × … × 0.80 (seven times) equals roughly 21%. Every match carries a real risk of elimination, and those risks stack up. By modelling at the match level, we see exactly how this uncertainty accumulates — and we can ask counterfactual questions like "what if Sinner draws Zverev in the quarter-finals?"

We use machine learning, not just Elo

The core predictor is XGBoost trained on 25 years of ATP matches. Pure Elo is elegant but ignores surface specialisation, recent form, head-to-head history, and age trajectories. XGBoost combines all of these into one calibrated probability estimate.

Design Choice We use a single model trained on all surfaces, not separate per-surface models. Surface specialisation is captured as a feature (surface_elo_diff) rather than a model split. This avoids training on too few clay-only samples to fit a reliable model.

We simulate the bracket, not just rank players

The natural temptation after computing clay ratings is to rank players and call the top one the favourite. But tournaments are not played on paper. Draw luck matters enormously. Monte Carlo simulation makes draw luck explicit: we run 20,000 independent bracket realisations and count how often each player wins. This approach is directly analogous to probabilistic Premier League season simulators — except instead of league positions, we track tournament rounds.

Step 1

1_fetch_data.py — 25 Years of ATP History

All match data comes from Jeff Sackmann's open tennis_atp repository on GitHub, which contains ATP match records dating back to 1968. We use 2000–2024, giving approximately 74,000 matches. Each year is a separate CSV file downloaded once and cached locally. 

python 1_fetch_data.py

# Downloading matches: 100%|██████| 25/25
# Loaded 74,906 matches (2000–2024)
# Loaded 55,193 players

Each record contains: winner, loser, surface (Hard / Clay / Grass), tournament level (Grand Slam, Masters, ATP 250…), the score, and both players' ATP rankings at match time.

Design Choice We start at year 2000, not earlier. Pre-2000 rankings are sparser and the player population differs significantly from the modern era. Twenty-five years gives ~75,000 matches — more than enough to train a robust model while keeping the data relevant.
Step 1b

fetch_recent.py — Supplementing with 2026 Clay Data

JeffSackmann's repository is updated manually and typically lags the current date by several weeks. For Roland Garros 2026, that means the clay build-up season — Monte-Carlo, Barcelona, Madrid, Rome — may only be partially available. We supplement it with tennis-data.co.uk, which publishes results within days of each match.

The critical difference is the player identifier. JeffSackmann uses stable numeric IDs like 207989 (Jannik Sinner). tennis-data uses abbreviated name strings: "Sinner J.". Before we can merge the two datasets, we must resolve every tennis-data name into the corresponding JeffSackmann ID.

The Name Resolution Problem

tennis-data names follow the pattern "Lastname F." — and sometimes "Lastname F.M." for double initials, or "Two Part Last C." for hyphenated surnames. The resolver handles all cases in three fallback layers: 

"Sinner J."        → parse → (last="sinner",  initial="j")  → exact match → 207989
"Cerundolo J.M."   → parse → (last="cerundolo", initial="j")  → exact match → 126610
"Ugo Carabelli C." → parse → (last="ugo carabelli", initial="c") → exact match
"Unknwon P."       → parse → fuzzy match on last name → best candidate or -1
Design Choice — Three-Layer Resolution 1. (last_name, first_initial) exact match — works for 95%+ of players.
2. Last-name-only — for surnames unique in the ATP, such as "Djokovic".
3. Fuzzy match (cutoff 0.75) — catches accents and slight spelling differences, e.g. "Müller" vs "Muller". Any match that fails all three layers is dropped with a warning rather than guessing — we never inject a wrong player ID into the training data.

Deduplication

Some matches appear in both sources — JeffSackmann may have the early rounds of Madrid while tennis-data has the full draw. After resolving IDs, we build a set of (winner_id, loser_id) pairs already present in the JeffSackmann data and exclude any tennis-data record that matches. This ensures each match is counted exactly once.

Step 2

2_build_features.py — Engineering Player Ratings

Raw match results are not useful to a model. We need features that capture each player's current strength just before each match is played. Four modules compute these features under a strict no look-ahead constraint: every value is computed using only matches that occurred before the one being labelled.

Elo Ratings

The classic chess rating system applied to tennis. Every player starts at 1500. After each match, the winner gains points proportional to the upset magnitude and the loser loses the same amount. We maintain two parallel Elo ratings per player: a global one and a surface-specific one.

P(A beats B) = 1 / ( 1 + 10 ( Elo_B − Elo_A ) / 400 )

The clay Elo is particularly important for Roland Garros — it accumulates evidence exclusively from clay-court matches and is the most stable long-run signal of clay court ability.

Head-to-Head Record

For each match we record how many times the winner has beaten the loser in all prior meetings. When two players have never met, this defaults to 0.5 — no information either way.

Rolling Surface Win Rate

Using a 365-day rolling window, we compute each player's win rate specifically on the current surface. A player on a 12-month hot streak on clay scores higher here than their long-run Elo suggests.

Recent Form

Win rate across the last 20 matches on any surface. This captures whether a player is currently healthy and confident — or going through a rough patch.

Step 3

3_train_model.py — Learning to Predict Matches

Dataset Symmetrisation

Each match record (winner, loser) is duplicated as (loser, winner) with the label flipped from 1 to 0. This doubles the training set from ~74,000 to ~148,000 examples and forces the model to learn symmetrically — it cannot exploit the fact that the "winner" column always won.

Train / Validation Split

We train on 2000–2021 and validate on 2022–2024. This is a temporal split, not a random split. Random splits would leak future information: a match from 2023 in the training set would have Elo values that already absorbed 2023 results.

XGBoost Model

500 gradient-boosted trees, depth 4, learning rate 0.03. The logistic objective means probabilities are meaningful — 0.65 means the model genuinely believes there is a 65% chance player 1 wins, not just a ranked score. 

python 3_train_model.py

# Train: 127,188 rows  (2000–2021)
# Val  :  17,958 rows  (2022–2024)
#
# Train accuracy : 0.686
# Val   accuracy : 0.658   ← generalises well
# Val   log-loss : 0.608
# Val   Brier    : 0.211   ← well-calibrated probabilities

Live Player State

After training, we replay all 74,000 matches chronologically through a PlayerStateTracker. This object maintains, for each of the 2,640 ATP players ever recorded, their current state: Elo ratings, ATP rank, clay win rate, form, and head-to-head records against every other player. This state is what powers predictions for any future matchup.

Design Choice The player state tracker is saved as data/processed/player_states.pkl. This means 4_simulate_rg2026.py does not need to replay 74,000 matches every time — it loads instantly and goes straight to the simulation. Each file acts as a checkpoint in the pipeline.
Step 4

4_simulate_rg2026.py — The Monte Carlo Tournament

Selecting the Field

We take the top 128 active ATP players by current ranking. These are the tournament entrants.

Clay Strength Scoring

Before building the draw, we rank all 128 players by a composite clay strength score. This is analogous to attack and defense parameters in Poisson football models — a single number that summarises how dangerous a player is on clay heading into the tournament.

Clay Strength  =  0.45 × Clay Elo (z-score)  +  0.30 × Clay Win Rate — 12 months (z-score)  +  0.25 × Overall Form — last 20 matches (z-score)
Design Choice Clay Elo gets the highest weight (45%) because it is the most stable signal over thousands of matches. Recent win rate (30%) captures current form on the surface. Overall form (25%) ensures a player on a hot streak is not penalised for playing on hard courts in the build-up. All three components are z-score normalised so they are on the same scale before combining.

Draw Construction

We replicate the official Grand Slam seeding rules:

  • Seed 1 → fixed position 1 (top of top half)
  • Seed 2 → fixed position 128 (bottom of bottom half)
  • Seeds 3–4 → drawn randomly to the two half-boundary slots
  • Seeds 5–8 → one drawn randomly per quarter
  • Seeds 9–16 and 17–32 → fill their respective sections
  • Remaining 96 unseeded players → placed randomly

Monte Carlo Simulation

We run 20,000 independent simulations of the full 128-player bracket. In each simulation:

  1. Start with the draw (bracket positions are the same in every simulation)
  2. For each first-round match, draw a random outcome from the match win probability
  3. The winner advances; repeat for Round 2 through the Final
  4. Record which player won the title, and which round each player reached

After 20,000 runs, each player's probability of reaching each round is the fraction of simulations where they reached it. With 20,000 runs, championship odds are stable to within roughly ±0.5 percentage points.

Design Choice — Match Probability Caching In a 128-player bracket, each unique pair of players is scored by the model at most once per simulation. But across 20,000 simulations the same pairings recur constantly. We cache all computed probabilities so each unique pair is evaluated only once. This reduces the per-simulation cost from 127 model evaluations to near zero after the first few hundred simulations, making 20,000 runs practical in minutes rather than hours.

What This Model Does Not Do

Being explicit about limitations matters as much as describing capabilities.

  • No injury or withdrawal modelling.
    If Sinner withdraws two days before the tournament, the model does not know. The draw would need to be rebuilt manually.
  • No match-level context.
    We do not model fatigue across consecutive matches, weather conditions, or scheduling gaps. Tennis is a physically brutal sport; back-to-back five-set matches are not captured.
  • No betting market calibration.
    Unlike Poisson football simulators, we do not blend our probabilities with bookmaker odds. The model is a pure statistical estimate from historical data — self-contained and fully interpretable, but not attempting to beat the market.
  • 2024 data cutoff.
    Training data ends at the 2024 season. To refresh the model, re-run 1_fetch_data.py once the 2025 CSV is available in the JeffSackmann repository, then re-run the full pipeline.

Function Reference

Below is a plain-language guide to every key function in the pipeline, organised by the file it lives in. You do not need to know Python to follow along — each entry explains what the function does, what you give it, and what you get back.

Data Collection  — src/data/

These functions are responsible for finding match results on the internet and loading them into the program. Think of them as the librarians that go out and collect all the books before the analysis can begin.

fetch.py fetch_matches(start_year, end_year, save_dir)

What it does: Goes to Jeff Sackmann's open GitHub archive — the most complete free ATP match database available — and downloads one results file per year, from start_year to end_year. Files that were already downloaded in a previous run are read from your hard drive instead of fetched again, so running the pipeline twice does not waste internet bandwidth. Each row in the output represents one completed ATP match: who played, who won, what the score was, what surface they played on, and what each player's ranking was at the time.

In Year range (e.g. 2000–2025) and a folder to save the files
Out A table of ~76,000 matches with columns for winner, loser, surface, date, ranking, and score
fetch.py fetch_players(save_dir)

What it does: Downloads the master player directory from the same JeffSackmann archive — a list of every ATP player ever recorded, with their numeric ID, first name, last name, nationality, and date of birth. This table is the "phone book" that other functions use to look up who a player is from their ID number. It is downloaded once and cached locally.

In A folder to save the file
Out A table of ~55,000 players with their IDs and biographical details
fetch_recent.py fetch_tennisdata_year(year, save_dir)

What it does: Downloads the current-year results file from tennis-data.co.uk, a separate website that publishes ATP match results within days of each tournament finishing. JeffSackmann's archive is updated manually and can lag by several weeks — so for the most recent clay-court build-up to Roland Garros (Monte-Carlo, Barcelona, Madrid, Rome) we need this supplementary source. The file is an Excel spreadsheet saved to your hard drive for reuse.

In The year (e.g. 2026) and a folder to save the Excel file
Out A raw table with column names specific to tennis-data.co.uk (Winner, Loser, WRank, Surface…)
fetch_recent.py build_name_index(players_df)

What it does: Creates two lookup dictionaries from the player master table, so that when we encounter a name like "Sinner J." from tennis-data, we can quickly find his JeffSackmann ID (207989). The primary dictionary maps a (last name, first initial) pair to an ID — the most precise lookup. The secondary dictionary maps just the last name to an ID, for rare surnames where the initial is not needed. Think of it as building two index cards — one more specific than the other — before you start looking players up.

In The players master table from fetch_players()
Out Two dictionaries: one keyed by (last name, initial) → ID, one by last name → ID
fetch_recent.py resolve_player_id(name, primary, secondary, all_last_names)

What it does: Translates a tennis-data name string like "Sinner J." into the numeric player ID used everywhere else in the project. It tries three approaches in order, stopping as soon as one succeeds:

1. Exact match on (last name + first initial) — works for 95%+ of players.
2. Last-name-only match — used when the surname is unique in the ATP, like "Djokovic".
3. Fuzzy match — handles typos and accent differences, e.g. "Müller" vs "Muller". If all three fail, the match is discarded rather than guessed — we never inject a wrong ID into the data.

In A raw name string from tennis-data and the two lookup dictionaries from build_name_index()
Out A numeric player ID, or −1 if no match could be found
fetch_recent.py map_tennisdata_to_sackmann(raw_df, players_df, surface_filter, complete_only)

What it does: Converts a tennis-data spreadsheet — which uses different column names, a different player naming convention, and different category labels — into the exact same format as the JeffSackmann data. This is the "translation layer" between the two data sources. It filters to only clay matches, drops unfinished matches (retirements, walkovers), resolves all player names to IDs, and renames every column so the result slots seamlessly into the historical dataset.

In Raw tennis-data table, the players master table, and optionally a surface filter (default: Clay)
Out A table in JeffSackmann format, ready to be appended to the historical data
fetch_recent.py fetch_supplement_clay(players_df, year, existing_matches, save_dir)

What it does: The single top-level function called by 1_fetch_data.py to top up the historical archive with the most recent clay results. It calls fetch_tennisdata_year() to download the file, map_tennisdata_to_sackmann() to convert it, and then removes any matches that are already present in the JeffSackmann data (deduplication) — so no match is counted twice even if both sources covered the same tournament.

In Players master table, the year to fetch, and the existing JeffSackmann matches (for deduplication)
Out A table of new clay matches not already in the historical archive
Data Cleaning  — src/data/process.py

Raw data from the internet is rarely tidy. This module standardises everything — dates, surface names, missing values — so all downstream code can rely on a consistent format.

process.py clean_matches(df)

What it does: Takes the raw combined match table and tidies it up. Specifically: it converts date strings like "20230529" into proper calendar dates; maps inconsistent surface labels to four canonical values (Hard, Clay, Grass, Carpet); fills in a placeholder rank (9999) for players whose ranking is missing; marks matches that ended in a retirement or walkover as "incomplete" so they can be excluded from training; sorts everything chronologically; and numbers every match with a unique index. Every other module in the pipeline depends on this function having been called first.

In The raw combined match table (from fetch_matches() or the merged dataset)
Out A cleaned, sorted table with proper date types, a completion flag, and a match index
Feature Engineering  — src/features/

"Features" are the statistics the model uses to decide who is likely to win a match. Raw data only tells us who played and who won — features transform that into meaningful numbers like "how good is this player on clay?" and "has he been on a hot streak lately?" Each module below computes one family of features by replaying every match in chronological order, using only information available before each match was played (no peeking into the future).

elo.py win_prob(rating_a, rating_b)

What it does: Given two Elo ratings, calculates the probability that player A beats player B in a single match. The formula is the classic Elo formula borrowed from chess: a player rated 100 points higher than their opponent is expected to win about 64% of the time; 200 points higher gives about 76%. This is the core mathematical engine behind all match predictions.

In Two Elo rating numbers (e.g. 1620.5 and 1480.0)
Out A probability between 0 and 1 (e.g. 0.72 means player A is 72% likely to win)
elo.py update_elo(r_winner, r_loser, k=32)

What it does: After a match is played, updates both players' Elo ratings. The winner gains points and the loser loses the same amount. The number of points exchanged depends on how surprising the result was — beating a much stronger opponent earns far more points than beating someone ranked far below you (exactly as in FIFA player ratings or chess rankings). The k parameter controls how quickly ratings react to new results; k=32 is the standard ATP value.

In The winner's current rating, the loser's current rating, and the k-factor
Out The new (higher) winner rating and the new (lower) loser rating
elo.py compute_elo_ratings(matches)

What it does: Walks through all 76,000 matches in date order, starting every player at the default rating of 1500, and applies update_elo() after every single match. It does this twice in parallel — once for a global rating (all surfaces combined) and once for a surface-specific rating (e.g. a separate clay Elo that only updates from clay matches). Crucially, for each match it records the ratings before the match is played, not after — this is what prevents the model from "looking into the future" during training.

In The full cleaned match table, sorted chronologically
Out The same table with four new columns added: global Elo and surface Elo for both the winner and loser
h2h.py compute_h2h(matches)

What it does: For every match in the dataset, counts how many times the winner had previously beaten the loser and vice versa — i.e., the head-to-head record going into that match. If they had never played before, the head-to-head win rate defaults to 0.5 (no information either way). This matters because some rivalries are consistently one-sided regardless of rankings — Djokovic vs Federer on clay, for instance.

In The full cleaned match table
Out The same table with new columns: head-to-head wins, losses, total meetings, and win rate for the winner
surface.py compute_surface_winrates(matches, window_days=365)

What it does: For each match, looks back over the previous 12 months and calculates each player's win percentage specifically on that match's surface. A player who has won 14 of their last 18 clay matches in the past year scores much higher here than their overall Elo would suggest. This rolling window means the stat always reflects current surface form rather than a career average, which is especially important because clay specialists often peak at different career stages than hard-court players.

In The match table and the window size (default: 365 days)
Out New columns added: 12-month win rate on the relevant surface, and number of matches in that window, for both players
form.py compute_recent_form(matches, window=20)

What it does: Calculates each player's win rate across their last 20 matches on any surface. Think of this as a "current form" indicator — it captures whether a player is playing well and injury-free right now, regardless of which surface they have been playing on. A player coming off a 15-5 stretch in their last 20 matches scores much higher than one who has won only 8 of 20. The window is a sliding one: it always looks at the last 20 matches, not a fixed calendar period.

In The match table and the window size (default: last 20 matches)
Out New columns added: recent form win rate and number of matches counted, for both players
builder.py build_features(matches, use_cache, cache_path)

What it does: The conductor function that runs all four feature modules (Elo → H2H → surface win rate → recent form) in sequence, each one adding new columns to the dataset. When all four are done it saves the enriched table to disk as a cache file. The next time you run the pipeline, if the cache file exists, this entire step is skipped and the saved file is loaded directly — turning a 5-minute computation into a 2-second file read.

In The cleaned match table from clean_matches()
Out The match table enriched with 10+ new feature columns, saved to data/processed/features.pkl
builder.py symmetrize(df)

What it does: Prepares the dataset for machine learning training by doubling every match. For each match record, it creates two rows: one from the winner's perspective (labelled "player 1 wins") and one from the loser's perspective (labelled "player 1 loses"). It then computes difference features — e.g. Elo of player 1 minus Elo of player 2 — which is what the model actually sees. This doubling prevents the model from learning a spurious shortcut ("the first column always wins") and ensures it learns symmetrically from both sides of every matchup.

In The enriched feature table (~76,000 rows)
Out A doubled training table (~152,000 rows) with difference features and a win/loss label
Model Training & Prediction  — src/models/

These are the functions that learn from the historical data and then apply those lessons to predict future matches.

train.py train_model(df, model_path, complete_only)

What it does: Trains the XGBoost prediction model on the historical symmetrised data. XGBoost is an algorithm that builds hundreds of simple decision trees in sequence — each one learning from the mistakes of the previous ones — and combines them into a single, highly accurate predictor. The model is trained only on matches from 2000–2022 (the training period) and then tested on 2023–present data it has never seen, to verify it generalises to new matches. The trained model is saved to disk and reused every time a match probability needs to be calculated.

In The symmetrised training data (years 2000–2022 only)
Out A trained model file saved to models/xgb_atp.joblib
predict.py PlayerStateTracker.process_all(matches)

What it does: Replays every match ever played — all 76,000 of them — in chronological order, updating a "live profile" for each player after every result. By the time it finishes, the tracker holds the most up-to-date stats for each of the 2,640+ ATP players ever recorded: their current Elo rating, clay Elo, ATP ranking, form over the last 20 matches, clay win rate over the last year, and head-to-head record against every opponent they have ever faced. These profiles are what power predictions for any future matchup. The entire state is saved to disk so this replay does not need to happen again for the simulation step.

In The full cleaned match history in chronological order
Out Internal player state dictionaries, saved to data/processed/player_states.pkl
predict.py MatchPredictor.predict(p1, p2, surface, h2h_winrate, h2h_total)

What it does: Given two player profiles and the surface being played on, returns the probability that player 1 wins the match. Internally it assembles the 11 features the model was trained on (Elo difference, surface Elo difference, ranking gap, head-to-head record, form difference, etc.), feeds them to the trained XGBoost model, and returns the output probability. This is the single function called tens of thousands of times during the Monte Carlo simulation — once per matchup per simulation.

In Two player profile dicts (from PlayerStateTracker), a surface string, and the head-to-head stats
Out A number between 0 and 1 — the probability that player 1 wins (e.g. 0.63)
Tournament Simulation  — src/rg/

These functions take the player profiles and match probabilities computed above and use them to simulate the entire Roland Garros tournament thousands of times.

clay_rating.py compute_clay_ratings(tracker, player_ids)

What it does: Produces a single "clay strength score" for each of the 128 players in the tournament field. Three statistics contribute to the score — clay Elo (45%), 12-month clay win rate (30%), and overall recent form (25%) — each one first standardised so they are on the same scale before being combined. The result tells us who deserves to be seeded where in the draw: the player at the top of the list gets Seed 1, the player at the bottom gets Seed 128. This composite approach is more reliable than using ATP ranking alone, because the ATP ranking reflects all surfaces, not just clay.

In The PlayerStateTracker with current player profiles, and the list of 128 player IDs in the field
Out A table of 128 players sorted by clay strength, with their scores and component values
draw.py create_rg_draw(player_ids, rng_seed)

What it does: Assigns all 128 players to their positions in the tournament bracket, following the official Grand Slam seeding rules that Roland Garros uses. Seeds 1 and 2 are placed at opposite ends of the draw so they can only meet in the final. Seeds 3 and 4 are drawn randomly to the semi-final boundary positions. Seeds 5–8 are drawn to quarter-final positions. The remaining 96 unseeded players are shuffled randomly into the remaining slots. Using the same rng_seed value every time ensures the bracket is identical across all 20,000 simulations — only the match outcomes vary, not the draw itself.

simulator.py RGSimulator.simulate(draw, n_sim, seed, verbose)

What it does: Runs the full tournament 20,000 times. In each simulation, it starts with the same bracket and plays every match by flipping a weighted coin — the weight being the probability from MatchPredictor.predict(). After each simulated final, the winner is recorded. After all 20,000 runs, each player's probability of reaching each round is simply the percentage of simulations where they reached it. To keep the computation fast, match probabilities are cached: if Sinner vs Zverev has been calculated once, the same number is reused every time that matchup appears in subsequent simulations without re-running the model.

In The bracket (128 player IDs in position order) and the number of simulations (default: 20,000)
Out A results table (128 rows) showing each player's probability of reaching each round — R128, R64, R32, R16, QF, SF, Final, and winning the title
